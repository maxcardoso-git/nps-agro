  <name>Task 1: Create backend Dockerfile and .dockerignore</name>
  <files>Dockerfile, .dockerignore</files>
  <action>
Create `Dockerfile` in the project root for the NestJS backend. Use node:22-alpine as base.

Multi-stage build with 3 stages:

**Stage 1 - deps:**
- FROM node:22-alpine AS deps
- WORKDIR /app
- COPY package.json package-lock.json ./
- RUN npm ci --ignore-scripts (production + dev deps needed for tsc build)

**Stage 2 - build:**
- FROM deps AS build
- WORKDIR /app
- COPY tsconfig.json tsconfig.build.json ./
- COPY src/ src/
- RUN npm run build (runs tsc -p tsconfig.build.json, outputs to dist/)

**Stage 3 - production:**
- FROM node:22-alpine AS production
- RUN apk add --no-cache postgresql-client (needed for run-migrations.sh which uses psql)
- WORKDIR /app
- COPY package.json package-lock.json ./
- RUN npm ci --omit=dev --ignore-scripts (production deps only)
- COPY --from=build /app/dist dist/
- COPY sql/ sql/
- COPY scripts/run-migrations.sh scripts/run-migrations.sh
- RUN chmod +x scripts/run-migrations.sh
- ENV NODE_ENV=production
- ENV PORT=3310
- EXPOSE 3310
- HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 CMD wget -qO- http://localhost:3310/health || exit 1
- CMD ["node", "dist/main.js"]

Do NOT install PM2 in the Docker image -- Docker replaces PM2 as process manager.
Do NOT copy .env files into the image -- env vars come from docker-compose.

Create `.dockerignore` in the project root:
```
node_modules
dist
.git
.gitignore
.planning
*.md
.env*
user-portal/node_modules
user-portal/.next
admin-console/node_modules
admin-console/.next
Analytics/node_modules
Analytics/.next
test
```

**Step B: Create Dockerfiles for all 3 frontends.**

All 3 follow the same multi-stage pattern but with different directory structures:
- user-portal: source files at top level (app/, components/, lib/, modules/, messages/, styles/, middleware.ts)
- admin-console: source files in src/ directory
- Analytics: source files in src/ directory

For each frontend, create a Dockerfile INSIDE the frontend directory (e.g., `user-portal/Dockerfile`).

Multi-stage build with 3 stages:

**Stage 1 - deps:**
- FROM node:22-alpine AS deps
- WORKDIR /app
- COPY package.json package-lock.json ./
- RUN npm ci --ignore-scripts

**Stage 2 - build:**
- FROM deps AS build
- WORKDIR /app
- COPY . .
- ARG NEXT_PUBLIC_API_URL
- ARG NEXT_PUBLIC_API_BASE_URL
- ARG NEXT_PUBLIC_DEFAULT_LOCALE=pt-BR
- RUN npm run build

Note on env vars: user-portal uses NEXT_PUBLIC_API_URL. admin-console and Analytics use NEXT_PUBLIC_API_BASE_URL. Pass the correct ARG for each. NEXT_PUBLIC_* vars must be available at BUILD TIME for Next.js to inline them.

**Stage 3 - production:**
- FROM node:22-alpine AS production
- WORKDIR /app
- ENV NODE_ENV=production
- COPY --from=build /app/public ./public
- COPY --from=build /app/.next/standalone ./
- COPY --from=build /app/.next/static ./.next/static
- ENV PORT=3000
- EXPOSE 3000
- HEALTHCHECK --interval=30s --timeout=5s --start-period=15s --retries=3 CMD wget -qO- http://localhost:3000 || exit 1
- CMD ["node", "server.js"]

Create a `.dockerignore` file in EACH frontend directory:
```
node_modules
.next
.git
*.md
```

IMPORTANT differences between the 3 frontends:
- user-portal has package-lock.json in the ROOT of the monorepo, NOT in user-portal/. The docker-compose build context will handle this. For user-portal/Dockerfile, if package-lock.json does not exist locally, use `npm install` instead of `npm ci`. Check if user-portal has its own package-lock.json first.
- admin-console and Analytics each have their own package-lock.json.

Actually, check: user-portal does NOT have its own package-lock.json (only the root does). admin-console and Analytics DO have their own. For user-portal, use `npm install --frozen-lockfile || npm install` as fallback, OR better: generate a lockfile during build. Simplest: just use `npm install` for user-portal since it has no lockfile. For admin-console and Analytics, use `npm ci`.
